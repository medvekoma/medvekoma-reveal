Architecture

Partitioning

~

  • Murmur3Partitioner: uniform distribution, Murmur hash
  • RandomPartitioner: uniform distribution, MD5 hash
  • ByteOrderedPartitioner: ordered distribution

Architecture

Replication

~

Replication factor: the number of copies for each row

Replication strategy:

  • SimpleStrategy: next nodes clockwise
  • NetworkTopologyStrategy: multiple racks and data centers

Architecture

Consistency Levels

~

Waiting for acknowledgement from how many rows

ONE, ALL, QUORUM, LOCAL QUORUM...

Architecture

Durability

~

Data is written to the commit log first

After recovering from a failure, commit logs are replayed

Architecture

Performance

~

Write performance - no read is needed

Read performance is also good - bloom filters

Architecture

take away

~

  • All nodes are equal (no single point of failure)
  • Gossip protocol
  • Tunable consistency
  • Updates are Inserts (no read is needed)

CQL

Schema again?

~

Being schema-free is nice, but it pushes schema definition to the business layer.

-

Schema is only metadata: ALTER TABLE is quick, null values are not stored.

-

Support for dynamic data through collections.

CQL

Think performance

~

  • WHERE: only by key or index
  • LIMIT clause - default is 10000
  • ORDER BY: very limited
  • No Joins, no GROUP BY

~

CQL

PRIMARY KEY

~

  • Partition key: row keys
  • Clustering key: column name prefix

CQL

UPSERT

~

Primary keys are mandatory, they identify rows

INSERT and UPDATE are equivalent


-- UserId is primary key
INSERT INTO User 
	(UserId, FirstName, LastName) 
VALUES 
	(1, 'Ada', 'Lovelace');
						

UPDATE User SET 
	FirstName = 'Ada', 
	LastName = 'Lovelace' 
WHERE 
	UserId = 1;

CQL

Lightweight transactions

~


INSERT INTO Users
	(UserName, PasswordHash, FirstName, LastName)
VALUES
	('Ada', '###', 'Ada', 'Lovelace')
IF NOT EXISTS;
						

UPDATE Users
	SET PasswordHash = 'newhash'
WHERE UserName = 'Ada'
	IF PasswordHash = 'oldhash';

CQL

Take away

~

  • SQL interface to Cassandra data structure
  • PRIMARY KEY ((PartitionKeys), ClusteringKeys)
  • Collection types: element storage
  • UPSERT - UPDATE and INSERT are equivalent
  • Writes without read
  • Think performance: LIMIT, WHERE

Data Modelling

Design by query

Secondary indexes - the bad

  • Maintained on each node

  • Results are combined

  • Several times slower than filtering by key

Data Modelling

Design by query

Secondary indexes - the good

Not as bad as it sounds. Measure!

Fast if data is previously filtered by key


CREATE TABLE track (
	track_id int,
	title text,
	album text,
	artist text,
	PRIMARY KEY(album, track_id)
);

CREATE INDEX tracks_by_artist ON track(artist);

SELECT * FROM track 
	WHERE album = 'Joshua Tree' 
	AND artist = 'U2';
